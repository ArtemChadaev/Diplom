# Архитектура Frontend — Feature-Sliced Design (FSD)

← [Назад к описанию Frontend](./README.md) | [Логика форм →](./forms-logic.md)

Проект построен на методологии **Feature-Sliced Design (FSD)** для четкого разделения ответственности, масштабируемости и изоляции бизнес-логики.

---

## 🏗️ Структура слоев (`src/`)

```
src/
├── app/        # Инициализация приложения, глобальные стили (index.css), провайдеры, роутер
├── pages/      # Страницы приложения (композиция виджетов и фич)
├── widgets/    # Самостоятельные крупные блоки (Header, Footer, Layout, сложные таблицы)
├── features/   # Пользовательские сценарии и бизнес-логика (формы авторизации, приемки, FEFO)
├── entities/   # Сущности предметной области (User, Product, Order, Zone) — DTO, API-запросы
└── shared/     # Переиспользуемый базовый код (UI-кит shadcn, cn(), utils, api-клиент)
```

> **Правило иерархии (FSD):** Модули могут импортировать только из нижележащих слоев. Импорты из вышележащих слоев строго запрещены.

---

## 🧠 Где живет бизнес-логика?

Вместо устаревшей структуры плоских хуков, логика распределена по архитектурным слайсам:

1. **`features/{feature-name}/model/`** — хуки управления состоянием форм (`react-hook-form`), Zod-схемы валидации, обработчики отправки данных (`onSubmit`).
2. **`features/{feature-name}/ui/`** — UI-компоненты конкретной фичи (например, `LoginForm.tsx`).
3. **`entities/{entity-name}/model/`** — DTO, типы сущностей, функции запросов к API (`fetch`), сторы/состояния.
4. **`entities/{entity-name}/ui/`** — визуальное представление сущности (например, карточка товара, строка таблицы).

## 📌 Правило колокации (Colocation)

Тесты размещаются **рядом** с тестируемым кодом (Go-style):

```
features/auth/login/
├── model/
│   ├── use-auth-login.ts
│   └── use-auth-login.test.ts    ← тест логики фичи
└── ui/
    ├── login-form.tsx
    └── login-form.test.tsx       ← тест UI-компонента
```

---

## ⚡ TDD (Test-Driven Development)

**Тест пишется до реализации хука или бизнес-логики.**

1. Опишите сценарий (Given/When/Then) в файле `.test.ts`.
2. Реализуйте логику в `model/`.
3. Убедитесь в успешном прохождении тестов.
