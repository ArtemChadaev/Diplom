# Testing Strategy & Security Checklist

← [Back to Main README](../../README.md) | [CI/CD Policy →](../other/workflow.md)

Данный документ описывает обязательные уровни тестирования, ключевые сценарии и инструменты для обеспечения надежности и безопасности системы.

---

## 1. Методология и Инструменты

Для проекта выбрана пирамида тестирования с упором на интеграционные тесты бизнес-логики.

| Уровень | Инструменты | Описание |
|---------|-------------|----------|
| **Unit (Backend)** | `go test`, `stretchr/testify` | Тестирование чистых функций, валидаторов и хэширования. |
| **Unit (Frontend)** | `Vitest`, `React Testing Library` | Тестирование UI-компонентов и Zod-схем. |
| **Integration** | `httptest`, `sqlmock`, `testcontainers` | Проверка API-эндпоинтов с участием реальной БД (или Docker). |
| **E2E (End-to-End)** | `Playwright` | Проверка полных пользовательских путей в браузере. |
| **Security** | `gosec`, `npm audit`, `OWASP ZAP` | Статический анализ кода и сканирование уязвимостей. |

## 2. Обязательные тестовые сценарии (Критические)

### 2.1. Аутентификация и Сессии
*   **TC-01: Верификация OTP.** Ввод верного кода $\rightarrow$ получение JWT. Ввод неверного кода 3 раза $\rightarrow$ блокировка попыток через Valkey.
*   **TC-02: Защита от кражи Refresh Token.** Попытка использовать один и тот же RT дважды $\rightarrow$ система должна отозвать все активные сессии пользователя (угроза компрометации).
*   **TC-03: RBAC (Доступы).** Попытка `Storekeeper` вызвать эндпоинт `/admin/users` $\rightarrow$ возврат `403 Forbidden`.

### 2.2. Учет серий и FEFO (Бизнес-логика)
*   **TC-04: Корректность FEFO.** Создать 3 серии с разными сроками годности. Оформить заказ. Проверить, что система зарезервировала серию с самой ранней датой истечения.
*   **TC-05: Выпуск из карантина.** Попытка отгрузить серию в статусе `quarantine` $\rightarrow$ ошибка. Попытка перевести в `available` пользователем без роли `qp` $\rightarrow$ ошибка.
*   **TC-06: Блокировка STOP-сигнала.** При добавлении серии в `recalled_batches`, все существующие остатки этой серии на складе должны автоматически получить статус `blocked`.

### 2.3. Складские операции
*   **TC-07: Журнал среды.** Валидация: ввод температуры 35°C в зону с режимом 2-8°C $\rightarrow$ запись создается, но система генерирует критический алерт (Warning).
*   **TC-08: Слепая инвентаризация.** При открытой сессии инвентаризации фронтенд не должен получать `expected_qty` из API до момента нажатия кнопки «Завершить».

## 3. Проверка безопасности (Security Checklist)

Обязательные проверки перед деплоем:

1.  **JWT Integrity:**
    *   Проверка подписи (HS256).
    *   Проверка `exp` (истечение) и `aud` (аудитория).
2.  **Input Validation:**
    *   Защита от XSS во всех текстовых полях карточки товара (особенно `description`).
    *   Защита от SQL Injection в поисковых запросах через GORM parameterized queries.
3.  **Cookies Security:**
    *   `refresh_token` должен иметь флаги: `HttpOnly`, `Secure`, `SameSite=Strict`.
4.  **Rate Limiting:**
    *   Ограничение на количество запросов `/auth/send-code` с одного IP (защита от спама/подбора).
5.  **Sensitive Data:**
    *   Отсутствие паролей и токенов в логах (`audit_logs` не должен содержать `token_hash`).

## 4. Регламент CI/CD тестирования

*   **Pre-commit:** Запуск `lint` (golangci-lint, eslint).
*   **Pull Request:** Запуск Unit и Integration тестов. Покрытие кода (coverage) не ниже 70%.
*   **Staging:** Запуск дымовых (Smoke) E2E тестов в Playwright перед релизом.
