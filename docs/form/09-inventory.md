# Форма 5.2: Инвентаризация

## Требования (пресловой-чеклист)

- [ ] Таблица `inventory_sessions`: `id`, `status: enum(draft, in_progress, completed, cancelled)`, `started_by`, `started_at`, `completed_at`, `zone_id` (null = весь склад)
- [ ] Таблица `inventory_items`: `id`, `session_id`, `product_id`, `batch_id`, `expected_qty` (из системы), `actual_qty` (вводит сотрудник), `discrepancy` (расчёт: actual - expected)
- [ ] Таблица `inventory_samples`: `id`, `session_id`, `product_id`, `batch_id`, `qty` — учёт контрольных образцов
- [ ] Таблица `price_groups`: `id`, `name`, `price_range_from`, `price_range_to` — ценовые группы для акта пересортицы
- [ ] Бэкенд: `POST /api/v1/inventory` — (admin/qp/warehouse_manager) начало инвентаризации
- [ ] Бэкенд: `GET /api/v1/inventory/:id` — детали сессии
- [ ] Бэкенд: `PUT /api/v1/inventory/:id/items/:item_id` — ввод фактического количества
- [ ] Бэкенд: `POST /api/v1/inventory/:id/complete` — (admin/qp) завершение, фиксирование расхождений
- [ ] Бэкенд: `POST /api/v1/inventory/:id/writeoff-act` — (admin) акт зачёта пересортицы
- [ ] **«Слепая инвентаризация»**: сотрудник не видит `expected_qty` до завершения сессии
- [ ] shadcn/ui: `npx shadcn@latest add progress alert-dialog`

## Интерфейс

### Лейаут страницы

- `/inventory` — список сессий инвентаризации
- `/inventory/:id` — страница активной инвентаризации (поиск товаров + ввод)
- `/inventory/:id/results` — итоги (только после завершения)

## Логика

### Слепая инвентаризация — механизм

На клиенте поле `expected_qty` приходит с сервера, но **не отображается** пока `session.status !== "completed"`.  
Технически данные можно не отправлять до завершения (privacy by design), но можно скрыть в UI — выбор архитектурный.  
Рекомендуется: сервер **не возвращает** `expected_qty` пока сессия не завершена.

```
GET /api/v1/inventory/:id/items (в статусе in_progress)
Ответ: Array<{
  id, product_id, mnn, sku, batch: { serial_number, expiry_date },
  actual_qty: number | null     ← уже введённое значение
  // expected_qty — НЕ ВОЗВРАЩАЕТСЯ
}>

GET /api/v1/inventory/:id/items (в статусе completed)
Ответ: Array<{
  ...то же,
  expected_qty: number,         ← теперь видно
  discrepancy: number           ← расчётное расхождение
}>
```

### Ввод позиции

```ts
const onSaveItem = async (sessionId: string, itemId: string, actualQty: number) => {
  await fetch(`/api/v1/inventory/${sessionId}/items/${itemId}`, {
    method: "PUT",
    body: JSON.stringify({ actual_qty: actualQty }),
  })
}
```

### Zod для ввода

```ts
const inventoryItemSchema = z.object({
  actual_qty: z.number().int().min(0, "Не может быть отрицательным"),
})
```

### Акт зачёта пересортицы (admin)

Логика: взаимозачёт излишков и недостач **внутри одной ценовой группы**.  
Форма (диалог):

```
Выбор ценовой группы: [Select]
Излишки для зачёта:
│ МНН │ Серия │ +Кол-во │ [☑] │
Недостачи для зачёта:
│ МНН │ Серия │ -Кол-во │ [☑] │
Итог: +3 vs -3 → зачтено 0
[Сформировать акт]
```

API: `POST /api/v1/inventory/:id/writeoff-act`  
Body: `{ surplus_item_ids: string[], deficit_item_ids: string[], price_group_id: string }`

### Контрольные образцы

Форма добавления образца — `<Dialog>` с полями: Товар (Combobox), Серия, Количество.  
`POST /api/v1/inventory/:id/samples`

## 🔐 Блок для администратора

| Действие | Кто |
|---------|-----|
| Начать инвентаризацию | `admin`, `qp`, `warehouse_manager` |
| Вводить фактическое количество | все имеющие доступ к складу |
| Завершить инвентаризацию | `admin`, `qp` |
| Акт зачёта пересортицы | **только `admin`** |
| Отменить инвентаризацию | **только `admin`** |

В итогах (`/inventory/:id/results`) блок «Акт зачёта пересортицы» скрыт от всех кроме `admin`.
